\hypertarget{md_Version_RELEASE_POLICY_autotoc_md234}{}\doxysection{Release Policy Standard}\label{md_Version_RELEASE_POLICY_autotoc_md234}
\href{\#table-of-contents}{\texttt{ Table of Contents}}

{\bfseries{\mbox{[}P\+R\+O\+J\+E\+C\+T-\/\+N\+A\+ME\mbox{]}}} follows {\itshape Semantic Versioning (Sem\+Ver)} whereby the {\itshape Version Identifier (V\+ID)} expresses a meaning about the characteristics of the identified software release, by encoding them into some ordered {\itshape Labels} where each of them specify a software release state feature.

Sem\+Ver is applicable from {\bfseries{\mbox{[}P\+R\+O\+J\+E\+C\+T-\/\+N\+A\+ME\mbox{]} v1.\+0.\+0}} onwards.

Sem\+Ver defines a software version given its {\itshape A\+P\+Is}, {\itshape Features}, {\itshape Bugs} and {\itshape Fixes}. A new software version contains {\itshape A\+PI}, {\itshape Features}, {\itshape Bug} or {\itshape Fix} changes respect to the lower version. The version changes even in case of pre-\/release or build data specification, but this changes do not imply any {\itshape A\+PI}, {\itshape Features} or {\itshape Bugs} level changes. However {\itshape A\+PI} level changes defines {\bfseries{M\+A\+J\+OR}} version upgrades, {\itshape Feature} level changes define {\bfseries{M\+I\+N\+OR}} version upgrades and {\itshape Bugs} level changes define {\bfseries{P\+A\+T\+CH}} version upgrades

Sem\+Ver\textquotesingle{}s V\+I\+Ds are essentially composed by a mandatory part, called {\bfseries{M\+A\+I\+N-\/\+V\+ID}}, and an optional part named {\bfseries{E\+X\+T-\/\+V\+ID}} in this format \mbox{[}N\+O\+R\+M-\/\+V\+ID\mbox{]}\mbox{[}E\+X\+T-\/\+V\+ID\mbox{]}.

\hypertarget{md_Version_RELEASE_POLICY_autotoc_md237}{}\doxysubsection{Versioning Precedence Rules}\label{md_Version_RELEASE_POLICY_autotoc_md237}
\href{\#table-of-contents}{\texttt{ Table of Contents}}

Precedence refers to how {\bfseries{V\+I\+Ds}} are compared to each other when ordered. The following precedence rules can be used to determine the precedence of a given {\bfseries{\mbox{[}M\+A\+I\+N-\/\+V\+ID\mbox{]}}} or {\bfseries{\mbox{[}E\+X\+T-\/\+V\+ID\mbox{]}}} respect to another\+:


\begin{DoxyItemize}
\item Precedence are calculated by {\bfseries{separating the version into major, minor, patch and pre-\/release identifiers}} in that order (build metadata does not figure into precedence).
\item {\bfseries{Precedence is determined by the first difference when comparing each of these identifiers from left to right}} as follows\+: major, minor, and patch versions are always compared numerically.
\begin{DoxyItemize}
\item Example\+: {\bfseries{1.\+0.\+0}} $<$ {\bfseries{2.\+0.\+0}} $<$ {\bfseries{2.\+1.\+0}} $<$ {\bfseries{2.\+1.\+1}}.
\end{DoxyItemize}
\item When {\bfseries{major, minor, and patch are equal}}, a {\bfseries{pre-\/release version has lower precedence than a normal version}}\+:
\begin{DoxyItemize}
\item Example\+: {\bfseries{1.\+0.\+0-\/alpha}} $<$ {\bfseries{1.\+0.\+0}}.
\end{DoxyItemize}
\item Precedence for two pre-\/release versions with the same major, minor, and patch version are determined by comparing each dot separated identifier from left to right until a difference is found as follows\+:
\begin{DoxyItemize}
\item {\bfseries{Identifiers consisting of only digits are compared numerically}}.
\item {\bfseries{Identifiers with letters or hyphens are compared lexically in A\+S\+C\+II sort order}}.
\item {\bfseries{Numeric identifiers always have lower precedence than non-\/numeric identifiers}}.
\item A {\bfseries{larger set of pre-\/release fields has a higher precedence than a smaller set}}, if all of the preceding identifiers are equal.
\begin{DoxyItemize}
\item Example\+: {\bfseries{1.\+0.\+0-\/alpha}} $<$ {\bfseries{1.\+0.\+0-\/alpha.\+1}} $<$ {\bfseries{1.\+0.\+0-\/alpha.\+beta}} $<$ {\bfseries{1.\+0.\+0-\/beta}} $<$ {\bfseries{1.\+0.\+0-\/beta.\+2}} $<$ {\bfseries{1.\+0.\+0-\/beta.\+11}} $<$ {\bfseries{1.\+0.\+0-\/rc.\+1}} $<$ {\bfseries{1.\+0.\+0}}.
\end{DoxyItemize}
\end{DoxyItemize}
\end{DoxyItemize}
